Most Interesting UX Decision

Progressive disclosure with "No Action Needed." Instead of forcing agents to fill every field before advancing, each repair group has a toggle: check it off or mark it "No Action Needed." Both count as 100% complete for that group.

This mirrors how a real walkthrough works — you often know a room is fine in 10 seconds. Forcing field entry would cause agents to skip the app entirely. The progress bar only reaches 100% if the agent has consciously touched every group, which builds confidence in the completeness of the estimate.

The second big call: a single scrollable page per section (not step-by-step screens). On mobile, agents can scan and tap fast without feeling like they're navigating a wizard.

2

What Is Broken or Fragile

Photo storage in localStorage. Images are base64-encoded and stored inline with project data. One project with 20 photos can exceed the 5 MB quota on older Android devices. The app handles this gracefully (strips photos, retries), but the user loses their photos silently.

Tesseract.js OCR loads a large WASM binary (~10 MB) on first use. On slow connections this feels broken. A loading spinner exists, but the experience is rough.

calcDeal / calcBRRRR previously wrote zero values back to the project whenever the Finance sub-tab wasn't active. This was patched (DOM presence guard), but the pattern is fragile — any future DOM restructure could re-introduce the bug.

Offline photo capture relies on the service worker caching the Tesseract WASM — this fails on first offline use if the user never ran OCR while online.

3

Creative Addition — Deal Intelligence Layer

The brief asked for a repair estimator. I extended it into a deal decision tool.

Deal Analyzer: Enter purchase price and ARV. Get projected profit, ROI, max offer (70% rule), holding cost estimate, and a plain-English deal score (Strong / Marginal / Risky).

BRRRR Calculator: Models the full refinance cycle — refi proceeds, cash left in, monthly cash flow, cash-on-cash return. Helps agents decide whether to flip or hold.

Risk Score: 1–5 score auto-computed from checked items — structural flags, missing systems assessments, deferred items. Surfaces properties that look cheap but carry hidden risk.

Why: Acquisition agents don't just need repair costs — they need to know if the deal works. Combining both in one walkthrough tool eliminates the spreadsheet-switching that kills field momentum.

4

What I'd Ship Next (2 More Days)

Day 1 — Contractor Bid Flow: The bid tracker exists but bid-vs-estimate variance comparison isn't surfaced as an alert. I'd add a "bids received" status to each section, auto-flag when the lowest bid exceeds the estimate by >20%, and make the Bids sub-tab the default after export so agents remember to update it.

Day 1 — Photo-to-IndexedDB migration: Move all photo storage from localStorage to IndexedDB. This eliminates the quota issue, unlocks unlimited photos, and makes the app genuinely production-ready for large properties.

Day 2 — Share link via URL: Encode the current project as a compressed base64 URL fragment. Agents could share a read-only estimate with a lender or partner by texting a link — no login, no server required.

Day 2 — Timeline → Calendar export: The timeline estimator generates week-by-week schedules. I'd add .ics export so agents can drop renovation milestones directly into Google Calendar.

⚡

Role of AI in Development

Claude (Anthropic) was used as the primary development partner throughout — architecture decisions, implementation, edge case analysis, and UX review. The most valuable use was iterative debugging: I'd describe a bug (e.g., "calcDeal writes zeroes when Finance tab isn't active") and Claude would diagnose the root cause and produce a targeted patch. AI dramatically compressed the feedback loop between "I noticed something broken" and "it's fixed." The code is mine in the sense that I directed every decision and reviewed every change — but I wouldn't have shipped this level of polish in this time frame without it.
